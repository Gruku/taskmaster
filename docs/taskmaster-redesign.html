Single Mode: Full Ceremony or Nothing

Taskmaster treats all work as if scope is settled. During exploration and rapid iteration, the pick → lock → worktree → implement → complete → log cycle adds friction without proportional value.

"The backlog starts to get in our way instead of helping us, especially when we're not stabilized on features"
High — wrong work done

No Target Verification

Tasks don't declare what files/systems they touch. AI picked a task, created a worktree, implemented changes — all against the wrong codebase. Nothing in the system flagged the mismatch.

"I spent time creating tasks, picking them, completing them, logging session summaries — all for work that turned out to be on the wrong target"
Medium — confusion

Epic vs Milestone Conflation

AI creates milestones per feature ("Auth milestone", "Dashboard milestone") instead of using them as temporal phases. The naming doesn't communicate the sequential/attention-filter purpose.

"AI would tend to create milestones per features, not for the time axis"
Medium — noise

Granularity Explosion

AI decomposes work into too many tasks. 25 tasks for one feature is a project plan, not a backlog. The overhead of managing each task exceeds the tracking value for a solo developer.

25 observatory-react-* tasks for a single feature redesign
Medium — drift

Stale Tasks Accumulate

When direction shifts, todo tasks become stale but look valid. Nobody cleans them up because cleanup is itself overhead. The backlog drifts from reality.

"Three layers of tracking for what's really one evolving thing"
Low — rigidity

"All Work Through Tasks" Is Too Strict

Some work is unplanned, exploratory, or cross-cutting. Forcing it through the task system adds overhead without adding tracking value. The system should observe, not gatekeep.

Why "Phase" instead of "Milestone"?

Attention filter

Phase 2 tasks are hidden from "next available" — reduces cognitive load for the human reviewing the board

Coarse dependency

Instead of wiring 8×12 individual task deps, say "Phase 2 collectively after Phase 1"

AI-proof naming

"Phase 1: Foundation" naturally reads as sequential. "Milestone: Foundation" gets confused with a feature goal

How it flows

Spike — "I need to figure out how the predictions page should work"
→ Named exploration, auto-logged, no overhead

Spike → Task — "Okay, I know what to build now. Break this into tasks."
→ Spike is completed, real tasks are created with anchors and deps

Untracked — "Hey fix this typo" / "Quick, the deploy is broken"
→ Just do it. start-session next time notes "3 commits outside tasks last session"

Spike Work Type

Lightweight exploration items: named in the backlog, auto-logged, no worktree/review/ceremony overhead. Promote to tasks when scope crystallizes.

Adds a type: spike field to backlog items. Spikes skip the pick-task ceremony (no lock, no worktree, no deps check) and end-session ceremony (auto-log from git diff instead of structured summary). start-session shows "Last time you were exploring: dash-spike" for continuity. 
- id: "dash-spike"
  title: "Figure out predictions page architecture"
  type: "spike"
  status: "in-progress"
  # No worktree, no review gate, no session ceremony
When a spike produces clarity, a backlog_promote_spike tool converts it to real tasks with anchors and dependencies.
High impact
Medium effort

Rename Milestone → Phase

Communicates sequential ordering and attention-filtering purpose. Prevents AI from creating feature-milestones.

Rename throughout: schema field, MCP tools (backlog_add_phase, backlog_advance_phase), skills, viewer. Update init skill to ask "what needs to be built first before you can do the rest?" instead of "define your milestones." Consider making target_date required to ground phases in time. 
phases:
  - id: "foundation"
    name: "Phase 1: Foundation"
    target_date: "2026-04-01"
  - id: "polish"
    name: "Phase 2: Polish"
    target_date: "2026-05-01"
Med impact
Low effort

Task Anchors

Tasks declare which files/directories/URLs they touch. pick-task verifies you're editing the right target before work begins.

New optional field anchors on tasks: 
- id: "dash-009"
  title: "Fix predictions table sorting"
  anchors:
    - "src/observatory/**"
    - "localhost:5173"
When pick-task runs, it shows: "This task is anchored to src/observatory/." If the AI starts editing files outside the anchor paths, the system warns: "You're editing predictions.html but this task is anchored to src/observatory/ — is that right?"

This would have caught the exact mistake from the conversation where hours were spent implementing against the wrong codebase.
High impact
Medium effort

Task Budget per Epic

Soft cap of 5-8 tasks per active epic. Forces coarser granularity — if you need 25 steps, that's a plan doc, not backlog items.

When backlog_add_task would push an active epic past 8 tasks, return a warning: "This epic has 8 tasks already. Consider grouping related work into fewer, coarser tasks — or link a plan document instead."

Not a hard block — just friction in the right direction. Tasks should be "things I might pick up in different sessions," not "steps within one session."
Med impact
Low effort

Staleness Decay

Todo tasks untouched for N sessions get auto-flagged as "stale?" during start-session. Quick y/n sweep keeps the backlog honest.

Track last_mentioned timestamp on tasks (updated when any tool references the task). During start-session, if a task has been in todo for 5+ sessions without mention, flag it:

"These tasks haven't been mentioned in 5+ sessions — still relevant?"
auth-007 Add SAML support — stale?
api-012 GraphQL migration — stale?

User can confirm (resets counter) or archive. Prevents the slow accumulation of zombie tasks.
Med impact
Medium effort

Auto-Summary (No Ceremony)

Replace structured end-session template with auto-generated git diff summary. Zero-effort session continuity.

Instead of the Done/Decisions/Issues template that requires user review and approval, automatically capture: 
### 2026-03-21 — auto
Files changed: 4 | +127 -43
Commits: "fix predictions sorting", "add filter dropdown"
Tasks touched: dash-009
User can optionally add context, but the default is zero-effort. The changelog becomes a git-log-plus rather than a report.

Full structured summaries still available for execution-mode tasks where decisions and issues matter.
Med impact
Low effort

Observe Untracked Work

Don't force all work through tasks. Let start-session note "4 commits outside any task last session" — informational, not judgmental.

During start-session, compare git log since last session against tasks that were in-progress. Report any commits that don't map to a tracked task:

"Last session: 4 commits outside tracked tasks — fix typo in README, bump deps, hotfix deploy script, experiment with new layout"

No judgment, no "you should have created a task." Just awareness. The backlog tracks planned work; git tracks actual work. They don't have to match.
Med impact
Low effort
